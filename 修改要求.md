# BetterEconomy 問題診斷與修復報告

日期：2026-07-02

---

## 一、程式碼審查發現的 Bug

### 🔴 高風險

**1. SQLite 未開啟 WAL 模式，可能造成餘額資料遺失**

- 位置：`SQLiteStorage.java` → `initialize()`
- 問題：HikariCP 連線池開了 5 條連線，但沒有設定 `PRAGMA journal_mode=WAL`。SQLite 預設不支援多連線同時寫入，高併發時（大量玩家同時交易）容易出現 `database is locked` 錯誤。由於寫入是非同步執行、失敗只會記錄 WARNING 不會重試，會造成**記憶體快取顯示的餘額和資料庫實際不一致**，重啟後玩家的錢可能與上次不同。
- 修法：連線建立後執行
  ```sql
  PRAGMA journal_mode=WAL;
  PRAGMA busy_timeout=5000;
  ```

**2. Vault 查詢可能造成主執行緒卡頓**

- 位置：`EconomyService.getBalance()` / `playerExists()`
- 問題：快取沒命中時會**同步**呼叫資料庫查詢。因為很多外掛（商店、GUI、領地系統）會在主執行緒直接呼叫 Vault API，一旦快取沒命中（尤其 MySQL 有網路延遲時），會直接卡住主執行緒，造成全伺服器頓一下（TPS 掉）。
- 修法：所有 Vault 查詢改為「先查快取，沒有就非同步補建，本次先回傳 0/上次已知值」，避免任何情況下在主執行緒進行阻塞式 DB 查詢。

### 🟡 中風險

**3. `/pay` 失敗時可能造成金錢憑空消失**

- 位置：`PayCommand.java` 第 149-154 行
- 問題：扣款成功但存款失敗時，程式會嘗試退款給付款者；但如果這次退款也失敗，完全沒有後續處理，錢就直接消失，雙方都不會被通知。
- 修法：退款失敗時要記錄成「待人工處理」的交易紀錄，並通知管理員/在 log 中標記為 SEVERE，而不是靜默失敗。

**4. 顏色代碼替換方式不安全**

- 位置：`ConfigManager.translateColorCodes()` 第 256-259 行
- 問題：直接把訊息裡所有 `&` 換成 `§`，不檢查後面接的字元是否為合法顏色代碼。若 `messages.yml` 裡有一般文字裡的 `&`（例如「購買 & 出售」），可能被誤判成格式碼、造成訊息顯示異常。
- 修法：改用 Bukkit 內建的 `ChatColor.translateAlternateColorCodes('&', message)`。

### 🟢 小問題 / 程式碼品質

**5. `getPlayerRank()` 對不存在的玩家算出錯誤排名**

- 位置：`MySQLStorage.java` / `SQLiteStorage.java` → `getPlayerRank()`
- 問題：SQL 子查詢對不存在的 UUID 會回傳 NULL，NULL 比較恆為 false，導致 `COUNT(*)` 算出 0，排名被誤判為 1（而不是「查無此人」）。
- 修法：查詢前先判斷 `playerExists()`，不存在就直接回傳 -1。

**6. 玩家改名後，資料庫 `username` 欄位不會更新**

- 位置：`addBalance()` / `subtractBalance()` / `setBalance()` 的 SQL 語句
- 問題：這幾個方法都沒有 `SET username = ?`，玩家改名後，baltop、交易紀錄裡顯示的還是舊名字。
- 修法：SQL 加上 `username = ?`，一併更新。

**7. `DatabaseManager.java` 內有死程式碼**

- 位置：`DatabaseManager` 的 `configManager` 欄位一直被設成 `null`，從未真正賦值使用。
- 影響：不影響功能，純粹是誤導閱讀程式碼的人，建議清除。

**8. 建立帳戶時沒有防併發鎖，會產生無害但擾人的錯誤紀錄**

- 位置：`EconomyService.createPlayerAccount()` → `MySQLStorage.createPlayerAccount()`
- 現象（實際日誌抓到）：
  ```
  [BetterEconomy] Error creating account for Server_Land
  java.sql.SQLIntegrityConstraintViolationException: Duplicate entry '...' for key 'PRIMARY'
  ```
- 原因：外部插件（本例為 Residence）在啟動時同時發出多個 Vault 請求，兩個非同步執行緒同時判斷「帳戶不存在」，同時嘗試 INSERT，其中一個因主鍵重複而失敗。結果無害（帳戶仍只會有一筆），但會洗版 log。
- 修法：建立帳戶前用 per-UUID 鎖（`EconomyService` 內已有的 `playerLocks` 機制）保護，或用 `INSERT ... ON DUPLICATE KEY UPDATE` / `INSERT OR IGNORE` 語法讓重複請求安全地被忽略而不是拋例外。

## 二、Server_Land 問題完整分析

### 現象
每次 `/eco wipe`（清空資料庫）後重啟伺服器，baltop 排行榜第一名都會自動多出一筆 `Server_Land - 100.00`。

### 根本原因（日誌已證實）

1. 伺服器上的 **Residence**（領地插件）在啟動時，會用一個虛擬名稱 **"Server_Land"** 代表「伺服器擁有的領地」，並透過 Vault 詢問這個名稱的經濟帳戶（`hasAccount` / `getBalance`）。
2. 這是業界常見手法：讓「伺服器」共用玩家帳戶的資料結構與程式碼，不用額外寫一套特殊邏輯。像 EssentialsX、Towny 等插件也有類似做法。
3. 一個**設計正確**的經濟插件（例如 CMI、EssentialsX）在查詢一個從未真正加入過的名稱時，應該老實回傳「無帳戶 / 0 元」，**不會**因為單純被查詢就自動開戶發錢。
4. 但 BetterEconomy 目前的 `EconomyService.getBalance(UUID)` 邏輯是：只要查詢一個不存在的帳戶，且 `economy.starting-balance.enabled: true`，就會**自動建立帳戶並發放起始金額（目前設定為 100）**——不論是真玩家加入，還是其他插件單純的查詢動作，一律視為「新玩家」。
5. 這就是為什麼 `/eco wipe` 把整張表清空後，Residence 一啟動又問了一次，BetterEconomy 又自動生了一筆 100 元出來。

### 為什麼以前沒發現？

舊版 wipe 邏輯是用「扣減/歸零」現有餘額，而不是真正 `DELETE` 整筆紀錄，所以 "Server_Land" 這筆帳戶資料庫裡其實一直都在，只是餘額被改掉、可能因為扣減邏輯有 bug 變成負數。因為紀錄「存在」，BetterEconomy 不會誤判成新帳戶，自然不會重新發錢——問題只是被舊 bug 意外蓋住了，兩個問題一直都在、只是互相抵消看不出來。改成真正 `DELETE` 之後，這個問題才第一次露出來。

### 為什麼 CMI 沒有這個現象？

不是 CMI 比較安全，而是**同一時間 Vault 只會有一個生效的經濟系統**，日誌證實：

```
Vault was found - Enabling capabilities. Economy: BetterEconomy
```

`VaultIntegration.java` 用 `ServicePriority.High` 註冊，蓋掉了 CMI 內建的（預設 Normal 優先度）經濟模組，所以 Residence 問的其實一直是 BetterEconomy，CMI 的經濟系統根本沒被使用到。如果換成只啟用 CMI 經濟，CMI 是否也會有一樣的問題，要看它內部有沒有做這個防呆（一般主流插件都有做，才沒被踩到雷）。

## 三、根本修復方案（比照 CMI / EssentialsX 的標準做法）

### 修改原則

把「自動開戶 + 發放起始金額」的觸發時機，從**任何查詢動作**，改成只有以下兩種情況才會觸發：

1. **玩家真正加入伺服器**（`PlayerJoinEvent`，即 `PlayerCacheRefreshListener`）
2. **真的發生金流異動**（`deposit()` / `withdraw()` / `setBalance()`，例如商店交易、指令發錢、扣租金）

而單純的**查詢類方法**（`getBalance()`、`hasAccount()` / `playerExists()`）在帳戶不存在時，一律直接回傳 `0` / `false`，**不再**順便建立帳戶、更不會發起始金額。

### 具體要修改的方法

| 方法 | 目前行為 | 修改後行為 |
|---|---|---|
| `EconomyService.getBalance(UUID)` | 查不到帳戶 → 自動開戶 + 給起始金額 | 查不到帳戶 → 直接回傳 `BigDecimal.ZERO`，不建立任何紀錄 |
| `EconomyService.refreshPlayerCache(UUID)` | 查不到帳戶 → 自動開戶 | 只有透過 `PlayerJoinEvent` 呼叫時才允許自動開戶 |
| `VaultEconomyImpl.hasAccount(...)` | 間接觸發自動開戶 | 純查詢，不觸發任何建立帳戶的動作 |
| `deposit()` / `withdraw()` / `setBalance()` | 內部 `ensurePlayerAccountExists()` 會自動開戶 | **維持不變**，這是正常、必要的行為（真的要給錢/扣錢時本來就該建立帳戶） |
| `PlayerCacheRefreshListener`（PlayerJoinEvent） | 呼叫 `refreshPlayerCache()` | **維持不變**，這是玩家真正加入時該有的自動開戶時機 |

### 影響評估

- ✅ 真正的玩家加入遊戲：完全不受影響，一樣正常拿到起始金額。
- ✅ 商店交易 / 發錢 / 扣款等會動到金額的操作：完全不受影響，一樣會自動開戶。
- ✅ 其他插件用假帳號名稱（如 "Server_Land"）單純查詢餘額：以後會得到正確的「0 元 / 無帳戶」回應，不會再無中生有拿到錢。
- ✅ 不只解決 "Server_Land"，未來任何插件用類似手法（"SERVER"、"Bank" 等虛擬名稱）查詢，也不會再誤觸發。
